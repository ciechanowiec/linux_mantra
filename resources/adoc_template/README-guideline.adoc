This guideline governs how written documents are produced. Its purpose is to make documents precise, complete, easy to follow, and consistent across authors and over time.

The guideline applies to non-fiction prose - reference documentation, design documents, runbooks, README files, protocol descriptions, and prose memos. It doesn't apply to fiction, marketing copy, or transient chat.

The guideline is written for authors of such prose, whether human or AI. It assumes the ability to write standard English and familiarity with the subject the author writes about. After reading, an author should be able to apply each rule while writing and to verify a finished document against the rules.

The guideline is organized in four parts. xref:#part-1[Part 1] states rules that apply to any non-fiction document. xref:#part-2[Part 2] adds rules for technical texts - documents whose subject is a technical system. xref:#part-3[Part 3] adds rules that apply when the author is an AI system. xref:#part-4[Part 4] adds rules for the AsciiDoc markup that carries the document.

[[part-1]]
== General Rules

This part states rules that apply to any non-fiction document, whatever its subject and markup. The parts that follow add rules for narrower contexts without overriding the rules stated here.

=== Audience, Purpose, and Scope

[#identify-audience]
==== Identify the Audience

A document names its primary reader at the beginning - the role, the assumed knowledge, and the decision the reader will make after reading. Rules that follow on style, depth, and vocabulary are calibrated to that reader. A document that doesn't identify its reader is calibrated to no one.

When the prompt or request names the reader, restate the reader's role and assumed knowledge in one sentence at the document's opening. This is calibration, not echoing - the document still owns its own framing. See xref:#dont-echo[].

[.bad-example]
.{bad-icon} Bad example
====
The Connection Pool Reference describes the connection pool.
====

[.good-example]
.{good-icon} Good example
====
The Connection Pool Reference is written for application engineers integrating against the database driver. It assumes familiarity with relational databases and connection lifecycle. It documents pool configuration, observable behavior, and tuning - enough to size and operate a pool against the supported drivers.
====

==== State the Purpose

A document states what it's for - the change in the reader's knowledge, capability, or behavior that the document is intended to produce. The purpose is named concretely enough that the reader can predict the kind of content that follows and judge, after reading, whether the document delivered.

[.bad-example]
.{bad-icon} Bad example
====
This document covers the rate limiter.
====

[.good-example]
.{good-icon} Good example
====
This document describes how the rate limiter behaves in production today and states the integration requirements that callers must satisfy. After reading it, an integrator should be able to send conforming requests, interpret the responses, and configure a quota for a new tenant.
====

==== Scope and Non-Scope

A document, and each major section, opens with a statement of what it covers. The scope statement bounds the topic precisely enough that the reader knows, before reading further, whether the document answers the question at hand. A non-scope statement is added only when a reasonable reader would otherwise expect the missing material to appear. A non-scope statement about material no one would expect is filler.

[.bad-example]
.{bad-icon} Bad example
====
(Document titled "The Order Service Reference" opens directly with: "An order is a record of a customer's intent to purchase one or more items...".) (No scope statement. The reader doesn't know whether the document covers the wire protocol, the storage format, the administrative interface, or all three.)
====

[.bad-example]
.{bad-icon} Bad example
====
(Document titled "The Order Service Wire Protocol" opens with: "This document covers the wire protocol of the Order Service. It doesn't cover unrelated subjects such as employee onboarding or office facilities.") (Scope is fine. The non-scope sentence addresses material no reader would have expected.)
====

[.good-example]
.{good-icon} Good example
====
(Document titled "The Order Service Reference" opens with: "This document specifies the wire protocol between client and server, the request and response formats, and the error model. The on-disk storage format and the replication protocol between server nodes are covered separately in the Storage Reference and the Replication Reference.") (Scope is stated. The non-scope is included because a reader of an "Order Service Reference" might reasonably expect storage and replication to be covered.)
====

=== Thought Structure

[#one-sentence-one-thought]
==== One Sentence, One Thought

A sentence carries exactly one assertion. If a sentence contains two independent claims joined by _and_, _but_, _while_, or a semicolon, split it. Compound sentences that link a cause to its consequence are acceptable when the link is the point. Sentences that staple two unrelated facts together aren't.

[.bad-example]
.{bad-icon} Bad example
====
The connection is established when the handshake completes, and a request is sent as a sequence of frames that are reassembled by the receiver in the order their stream identifiers appear.
====

[.good-example]
.{good-icon} Good example
====
The connection is established when the handshake completes. A request is sent as a sequence of frames. The receiver reassembles the frames in the order their stream identifiers appear.
====

[#one-paragraph-one-topic]
==== One Paragraph, One Topic

A paragraph develops one thought: a claim followed by its elaboration, qualifications, and consequences. When the topic shifts, the paragraph ends. A paragraph that introduces a second, parallel topic must be split, even when the two topics are related.

[.bad-example]
.{bad-icon} Bad example
====
A transaction groups a series of operations into a single atomic unit and is started by calling `begin`. Indexes accelerate query execution by storing precomputed lookup structures and are maintained automatically by the database.
====

[.good-example]
.{good-icon} Good example
====
A transaction groups a series of operations into a single atomic unit. The transaction is started by calling `begin` and is concluded by either `commit` or `rollback`. Until the transaction is concluded, its changes are visible only to operations performed within the same transaction.

An index is a precomputed lookup structure that accelerates query execution. The database maintains an index automatically: every insert, update, and delete on the indexed table updates the index in the same transaction as the data change.
====

==== No Restatement

A thought appears in exactly one place in the document. Two consecutive sentences that say the same thing in different words are a defect. A paragraph that ends by summarizing itself is a defect. When you find yourself rephrasing, delete the weaker version.

[.bad-example]
.{bad-icon} Bad example
====
The cache is invalidated when the source changes. Whenever the underlying data is modified, the cached copy is no longer valid and is therefore removed.
====

[.good-example]
.{good-icon} Good example
====
The cache entry for a record is invalidated when the record's `last++_++modified` timestamp changes.
====

==== Razor Sentences

ifdef::backend-html5[]
++++
<!-- vale Google.We = NO -->
++++
endif::[]
Every sentence does work that no other sentence in the document does. If a sentence can be deleted without losing information, delete it. The usual offenders are sentences that announce upcoming material, sentences that summarize what was just said, and sentences that exist only to introduce a list. "In this section, we will discuss..." announces upcoming material. "There are several ways to do this:" introduces a list and can be deleted along with its colon.

[.bad-example]
.{bad-icon} Bad example
====
There are several important things to consider when configuring the load balancer. Let us look at them. The first thing is the algorithm. The algorithm controls how requests are distributed across backends.
====
ifdef::backend-html5[]
++++
<!-- vale Google.We = YES -->
++++
endif::[]

[.good-example]
.{good-icon} Good example
====
The load balancer's algorithm controls how requests are distributed across backends. The supported algorithms are round-robin, least-connections, and consistent-hash. The default is round-robin.
====

==== Forward Order

Information flows from the general to the specific, from the prerequisite to the dependent, and from the cause to the consequence. The reader should never need a fact that the document hasn't yet introduced. When a concept is needed before its full definition, give a short gloss at first mention and link to the full definition.

[.bad-example]
.{bad-icon} Bad example
====
The `onMessage` callback is invoked after the channel has joined the topic. A channel is a logical connection identified by a name and bound to a single topic.
====

[.good-example]
.{good-icon} Good example
====
A channel is a logical connection identified by a name and bound to a single topic (see §3.2). When a channel joins a topic, it begins to receive messages published to that topic. Each message is delivered to the channel's `onMessage` callback.
====

==== Affirmative Before Negative

A claim states what something is or does before it states what it isn't or doesn't do. The affirmative gives the reader the fact to hold. The negation then refines the fact by ruling out an alternative the reader might otherwise assume. A negation placed first forces the reader to carry a list of exclusions with nothing yet to attach them to.

[.bad-example]
.{bad-icon} Bad example
====
Setting `cache.ttl` doesn't change the maximum number of entries or the eviction order. It controls only how long an entry remains valid after it's written.
====

[.good-example]
.{good-icon} Good example
====
Setting `cache.ttl` controls only how long an entry remains valid after it's written. It doesn't change the maximum number of entries or the eviction order.
====

==== Headings as Promises

A heading announces exactly what the section contains. The section delivers exactly that, no more and no less. A section titled "Authentication" describes authentication. It doesn't also describe authorization, even briefly. Material that doesn't belong under the heading goes into a separate section.

[.bad-example]
.{bad-icon} Bad example
====
Heading: "Connection Pooling." Body: discusses pooling, then transaction isolation, then a digression on prepared statements.
====

[.good-example]
.{good-icon} Good example
====
Heading: "Connection Pooling." Body: pool sizing, acquisition, release, leak detection, and the lifecycle of a pooled connection. Transaction isolation is documented in a separate section, with a cross-reference where pool behavior interacts with isolation.
====

=== Terminology and Precision

==== Strict Nomenclature

A concept is named once and the name is used everywhere. The name isn't varied for stylistic relief. Treat terms like identifiers in source code: renaming them mid-document is a defect that breaks the reader's ability to track the same thing across sentences.

This applies with equal force to operations. _Create_, _insert_, _add_, _register_, _publish_, and _submit_ aren't synonyms. Each names a specific operation in most systems. When the system distinguishes them, the document distinguishes them. When the system doesn't, pick one and use only that one.

[.bad-example]
.{bad-icon} Bad example
====
The handler registers itself with the broker. Once the listener is active, the consumer dispatches incoming events to subscribers. (Three names. One concept.)
====

[.good-example]
.{good-icon} Good example
====
The handler registers itself with the broker. Once the handler is active, it dispatches incoming events to subscribers.
====

[.bad-example]
.{bad-icon} Bad example
====
You can add a record to the table, then submit the changes by committing the transaction.
====

[.good-example]
.{good-icon} Good example
====
A record is inserted into the table by `INSERT INTO`. The change becomes durable when the enclosing transaction is committed.
====

[#defined-terms]
==== Defined Terms

A term that has a specific meaning in the document is defined the first time it's used in a normative sense. The definition takes the form _X is Y that Z_. _Y_ is the genus - the broader category. _Z_ is the differentia - what distinguishes _X_ from other members of _Y_. After the definition, the term is used without redefinition.

ifdef::backend-html5[]
++++
<!-- vale Google.Contractions = NO -->
++++
endif::[]
[.bad-example]
.{bad-icon} Bad example
====
A token is something the client sends to prove who it is.
====
ifdef::backend-html5[]
++++
<!-- vale Google.Contractions = YES -->
++++
endif::[]

[.good-example]
.{good-icon} Good example
====
A _token_ is an opaque string issued by the authentication server that the client presents to prove its identity to a resource server. A token has an issuer, a subject, an expiration time, and a set of scopes that constrain the operations the bearer may perform.
====

==== Concrete Vocabulary

Use words that two competent practitioners would interpret the same way. When two reasonable readers would disagree on what a word means in context, the word is too vague. Replace abstract qualifiers - _flexible_, _robust_, _scalable_, _modern_, _enterprise-grade_, _seamless_, _intelligent_, etc. - with the specific properties they hide.

[.bad-example]
.{bad-icon} Bad example
====
The platform is flexible and supports seamless integration with modern enterprise systems.
====

[.good-example]
.{good-icon} Good example
====
A new data source is added by implementing the `DataSource` interface and registering it through the plugin manifest. The platform discovers the implementation at startup. Existing sources don't need to be modified or recompiled, and existing queries continue to run unchanged.
====

==== Plain Names for Plain Things

Prefer the older, well-understood term over a newer marketing-laden one when both denote the same thing. Use the fancier term only when it carries information the plain term lacks. The test is whether replacing the fancier term with the plain one would lose meaning. When it would not, the fancier term was decoration.

[.bad-example]
.{bad-icon} Bad example
====
The system leverages a cloud-native, distributed compute fabric to deliver intelligent responses.
====

[.good-example]
.{good-icon} Good example
====
Each request is processed by one of several stateless servers behind a load balancer. The server consults a cache, then the database, then returns the response.
====

Further pairs where the plain form is usually correct:

* _server_ over _compute node_
* _file_ over _blob_ (when it's a file)
* _call_ over _invocation_ (when it's an ordinary function call)
* _list_ over _collection_ (when the order matters)
* _user_ over _actor_ (when it's a person)

==== Acronyms and Abbreviations

An acronym is expanded the first time it appears in the document and the first time it appears in each major section, in the form _Expanded Form (ACRONYM)_. After expansion, the acronym is used. Acronyms more recognizable than their expansions - HTTP, URL, JSON, XML, SQL, TCP - need not be expanded, except where the expansion adds information the reader needs.

++++
<!-- vale Google.Acronyms = NO -->
++++
endif::[]
[.bad-example]
.{bad-icon} Bad example
====
The CRDT replica synchronizes via the SWIM protocol over QUIC. (Three acronyms. The reader may know none.)
====
ifdef::backend-html5[]
++++
<!-- vale Google.Acronyms = YES -->
++++

[.good-example]
.{good-icon} Good example
====
Each replica is a Conflict-free Replicated Data Type (CRDT), so concurrent updates from different replicas can be merged without coordination. Replicas discover each other and detect failures using the Scalable Weakly Consistent Infection-style Membership (SWIM) protocol, which runs over Quick UDP Internet Connections (QUIC).
====

=== Context and Framing

==== Set the Stage

Before describing a topic in detail, locate it in the broader landscape. Name the category it belongs to, the problem it solves, and how it relates to adjacent concepts the reader is likely to know. The introduction need not be long. It must give the reader a place to put the new information.

[.bad-example]
.{bad-icon} Bad example
====
The query planner uses a cost-based optimizer with histograms.
====

[.good-example]
.{good-icon} Good example
====
A relational database receives queries in a declarative language (SQL) and must decide how to execute them. This is the job of the _query planner_: it transforms a SQL statement into an executable plan - a tree of operations such as scans, joins, and aggregations.

The planner described in this document is _cost-based_: it enumerates candidate plans and chooses the one with the lowest estimated cost. Cost is estimated using _histograms_ - summaries of column value distributions, refreshed periodically - which let the planner predict how many rows each operation will produce.
====

==== Outer Topics in Outline Only

When a topic touches an outer topic, summarize the outer topic in just enough detail for the current discussion. Don't reproduce its full documentation. Reference the authoritative source. The summary covers the slice of the outer topic that the current text depends on, no more.

[.bad-example]
.{bad-icon} Bad example
====
A section on rate limiting that opens with five paragraphs explaining what HTTP is, what status codes are, and how clients and servers communicate.
====

[.good-example]
.{good-icon} Good example
====
Rate limiting is the mechanism by which an HTTP server protects itself from excessive requests by a single client. When a client exceeds its quota, the server responds with status `429 Too Many Requests` and a `Retry-After` header indicating when the client may try again. The full semantics of HTTP status codes and headers are defined in RFC 9110.
====

==== Why Before How

Before describing a mechanism, state what the mechanism is for. The reader needs the purpose to evaluate the mechanism. _Why_ takes one or two sentences. It doesn't become a sales pitch.

[.bad-example]
.{bad-icon} Bad example
====
The deduplication subsystem maintains a Bloom filter of recent message IDs and consults it on each incoming message.
====

[.good-example]
.{good-icon} Good example
====
The same message can arrive more than once: producers retry on timeout, and the network can reorder and duplicate packets. To deliver each message to the application at most once, the receiver deduplicates incoming messages by ID. A Bloom filter of recent IDs serves as the first check. On a positive match, the receiver consults the durable index to confirm.
====

=== Concreteness and Examples

[#show-then-tell]
==== Show, Then Tell

Where a concept can be illustrated - by an example, a sample, a worked-out case, a quotation - show it. The illustration follows the definition. It doesn't replace it. Definitions establish meaning. Illustrations confirm it. When the concept has interesting boundaries - empty inputs, missing inputs, partial information, edge cases - illustrate at least one boundary alongside the typical case. Misreadings tend to arise at boundaries.

[.bad-example]
.{bad-icon} Bad example
====
A query parameter is appended to the URL after the path and is used to pass additional data to the server.
====

[.good-example]
.{good-icon} Good example
====
A query parameter is appended to the URL after the path, separated from the path by `?` and from other parameters by `&`. The following URL has two query parameters, `q` and `page`:

....
https://example.com/search?q=postgres+index&page=2
....

A parameter without a value is permitted. Its value is the empty string. A parameter that appears more than once produces a list of values whose order matches the order in the URL:

....
https://example.com/search?tag=db&tag=index&debug
....
====

[#realistic-examples]
==== Realistic Examples

Examples use names, values, and scenarios that resemble real usage: `OrderService`, `customer++_++id`, `/var/log/app.log`. Avoid `Foo`, `Bar`, `xxx`, and `myThing` except in the most abstract illustrations. Realistic examples carry information about typical usage. Placeholder examples don't.

[.bad-example]
.{bad-icon} Bad example
====
Acme Corp. agrees to provide widgets to Beta Ltd. on a date in the future.
====

[.good-example]
.{good-icon} Good example
====
Supplier (Acme Corp., a Delaware corporation, EIN 99-1234567) shall deliver 1,200 sensor modules (part number SNS-V2-A) to Customer (Beta Ltd., a Polish limited liability company, KRS 0000123456) at Mostowa 12, 50-200 Wrocław, Poland, on or before 2026-09-30.

=== Language and Style

==== Active Voice and Named Actor

A sentence names the actor that performs the action. Passive voice is reserved for cases where the actor is genuinely unknown, irrelevant, or the topic of the sentence. "The request is rejected" is acceptable when the topic is the request, not the rejecter.

[.bad-example]
.{bad-icon} Bad example
====
The configuration is loaded and the cache is populated. (Who loads it? Who populates the cache?)
====

[.good-example]
.{good-icon} Good example
====
On startup, the configuration loader reads `config.yaml` from the working directory and populates the in-memory cache.
====

==== Direct Verbs

Replace nominalizations with the underlying verbs. _Perform an analysis of_ becomes _analyze_. _Make a decision about_ becomes _decide_. _Provide support for_ becomes _support_. The shorter form is also more precise, because the verb carries the action that the nominalization had abstracted away.

[.bad-example]
.{bad-icon} Bad example
====
The framework performs the resolution of dependencies and provides notification of state changes.
====

[.good-example]
.{good-icon} Good example
====
The framework resolves dependencies and notifies registered listeners of state changes.
====

==== Avoid Parentheses and Semicolons

A parenthetical buries content the reader still has to process. When the content matters, write it as a separate sentence. When it doesn't matter, delete it. A semicolon joins two independent clauses into one sentence, which xref:#one-sentence-one-thought[] forbids - split the clauses into two sentences.

ifdef::backend-html5[]
++++
<!-- vale Google.Semicolons = NO -->
++++
endif::[]
[.bad-example]
.{bad-icon} Bad example
====
The connection pool (default size 10) is configured via `pool.size`; the value is read at startup.
====
ifdef::backend-html5[]
++++
<!-- vale Google.Semicolons = YES -->
++++
endif::[]

[.good-example]
.{good-icon} Good example
====
The connection pool is configured via `pool.size`. The default size is 10. The value is read at startup.
====

==== Natural Phrasing

ifdef::backend-html5[]
++++
<!-- vale write-good.TooWordy = NO -->
<!-- vale Google.We = NO -->
++++
endif::[]
Write the way an experienced practitioner would write a careful internal document. Avoid phrases - _herein_, _aforementioned_, _leverage_, _utilize_, _whilst_, _thereof_, etc. - that a practitioner would not say aloud. Avoid phrases - _super easy_, _just_, _simply_, _merely_, etc. - that minimize the reader's task.

[.bad-example]
.{bad-icon} Bad example
====
Herein, the aforementioned mechanism shall be utilized to leverage the synergies between the modules.
====
ifdef::backend-html5[]
++++
<!-- vale write-good.TooWordy = YES -->
<!-- vale Google.We = YES -->
++++
endif::[]

[.good-example]
.{good-icon} Good example
====
The two modules communicate through the event bus defined in §3.1. The producer publishes events. The consumer subscribes and processes them.
====

==== Specific Numbers

When a quantity matters, give the number, the unit, and the conditions under which it holds. _Fast_, _small_, _large_, _many_, and _most_ aren't numbers. When the number is unknown, say so and give a range or an order of magnitude.

[.bad-example]
.{bad-icon} Bad example
====
The cache holds many entries and evicts old ones quickly.
====

ifdef::backend-html5[]
++++
<!-- vale Google.LyHyphens = NO -->
++++
endif::[]
[.good-example]
.{good-icon} Good example
====
The cache holds up to 10,000 entries by default. When the cache is full, the least-recently-used entry is evicted on each insertion. Eviction is performed synchronously and typically completes in under 50 µs on commodity hardware.
====
ifdef::backend-html5[]
++++
<!-- vale Google.LyHyphens = YES -->
++++
endif::[]

[#no-hedging]
==== No Hedging

ifdef::backend-html5[]
++++
<!-- vale write-good.Weasel = NO -->
++++
endif::[]
Avoid hedges that weaken claims without adding information: _somewhat_, _fairly_, _quite_, _rather_, _generally_, _usually_ (unless followed by the exception), _kind of_, _a bit_, etc. When a claim has exceptions, state them. When a claim is uncertain, say what the uncertainty is.

[.bad-example]
.{bad-icon} Bad example
====
The operation is generally pretty fast and usually completes fairly quickly.
====
ifdef::backend-html5[]
++++
<!-- vale write-good.Weasel = YES -->
++++
endif::[]

[.good-example]
.{good-icon} Good example
====
The operation completes in under 10 ms for inputs of fewer than 1 million elements. For larger inputs, runtime grows linearly with input size. See §7.4 for benchmarks.
====

[#no-false-universals]
==== No False Universals

A claim in the form _X is Y_ or _X does Z_ asserts that the claim holds in every case. Write that form only when no case is excluded. When the behavior is a possibility, a permission, or one case among several, write the modality that holds: _can_, _may_, or the condition under which the claim is true. The rule mirrors xref:#no-hedging[]: a hedge understates a certain claim, and a false universal overstates an uncertain one.

[.bad-example]
.{bad-icon} Bad example
====
A query uses an index. The response is served from the cache. (Both sentences state one possibility as the rule. A query may run without an index. The response comes from the cache only on a hit.)
====

[.good-example]
.{good-icon} Good example
====
A query can use an index. The response is served from the cache on a hit and from the database on a miss.
====

==== No Patronizing

ifdef::backend-html5[]
++++
<!-- vale Google.Contractions = NO -->
++++
endif::[]
Don't tell the reader that something is _easy_, _simple_, _straightforward_, _obvious_, or _trivial_. When it is, the reader can see for themselves. When it isn't, the label rings false. State the steps. Let the reader judge the difficulty.
ifdef::backend-html5[]
++++
<!-- vale Google.Contractions = YES -->
++++
endif::[]

[.bad-example]
.{bad-icon} Bad example
====
Configuring the resolver is straightforward - just provide a simple hook implementation.
====

[.good-example]
.{good-icon} Good example
====
To customize resolution, implement the `ResolutionHook` interface and register the implementation through the configuration file under `resolution.hooks`. The framework consults registered hooks in declaration order. The first hook that returns a non-null result determines the outcome.
====

==== Tense and Mood

Use the present tense for behavior that holds whenever the system is in the relevant state: "The framework calls the start method...". Use the future tense only for actions that genuinely lie in the future from the reader's reference point - for instance, behavior of a planned but not yet released version. Use the imperative for instructions to the reader: "Register the listener before...".

[.bad-example]
.{bad-icon} Bad example
====
When the connection will be established, the server will be sending a greeting.
====

[.good-example]
.{good-icon} Good example
====
When the connection is established, the server sends a greeting frame containing its protocol version and supported extensions.
====

==== Person

ifdef::backend-html5[]
++++
<!-- vale Google.We = NO -->
++++
endif::[]
Use the third person for descriptions of the system. Use the second person ("you") for instructions directed at the reader. Avoid the first-person plural ("we") except in introductions or rationale, where it refers to the document's authors. Don't switch among _you_, _we_, _the user_, and _one_ within a section.

[.bad-example]
.{bad-icon} Bad example
====
We register the handler. The user can then look it up. One must be careful to unregister it on shutdown. You should also handle exceptions.
====
ifdef::backend-html5[]
++++
<!-- vale Google.We = YES -->
++++
endif::[]

[.good-example]
.{good-icon} Good example
====
Register the handler during initialization and store the returned registration. During shutdown, call `unregister` on the stored registration. When initialization throws, the framework doesn't invoke shutdown, so any partial registration must be cleaned up before the throw.
====

[#pronoun-antecedents]
==== Pronoun Antecedents

Every pronoun - _it_, _this_, _they_, _that_ - has a single, unambiguous antecedent in the same sentence or the immediately preceding sentence. When the candidate antecedents are two or more, replace the pronoun with the name. _This_ and _that_ at the start of a sentence are followed by the noun they refer to - _this change_, _that release_ - when the antecedent isn't a single noun.

[.bad-example]
.{bad-icon} Bad example
====
The handler calls the resolver, which logs the request, and it returns the result. (Does _it_ refer to the handler or the resolver?)
====

[.good-example]
.{good-icon} Good example
====
The handler calls the resolver. The resolver logs the request and returns the result to the handler.
====

[#inclusive-language]
==== Inclusive Language

Name actors and examples without invoking gender, ethnicity, or other identity attributes unless they're part of the topic. Use _they_ as a singular pronoun for a person of unspecified gender. Use names from a variety of traditions in examples, not from a single naming convention. Avoid metaphors that depend on culturally specific knowledge for a reader who may not share it. Replace terms whose origin demeans a group. The established replacements - _allowlist_ / _blocklist_, _primary_ / _replica_, _main_ branch, etc. - communicate at least as well.

[.bad-example]
.{bad-icon} Bad example
====
When the user clicks Submit, he is shown the receipt. If his card is declined, ask him to retry.
====

[.good-example]
.{good-icon} Good example
====
When the user clicks Submit, the receipt is shown. If the card is declined, the user is prompted to retry.
====

=== Anti-Fluff

[#no-marketing]
==== No Marketing

Documentation doesn't sell. Remove claims of being _cutting-edge_, _next-generation_, _industry-leading_, _world-class_, _blazing fast_, _rock-solid_, _battle-tested_, etc. Replace them with what they were trying to claim, in concrete terms, or remove the sentence entirely. When a document does argue for a position, it advances claims supported by evidence, not by superlatives.

[.bad-example]
.{bad-icon} Bad example
====
The blazing-fast, industry-leading database delivers world-class performance for the most demanding workloads.
====

[.good-example]
.{good-icon} Good example
====
The database sustains 50,000 reads per second and 8,000 writes per second on the reference hardware described in Appendix B, with p99 read latency under 5 ms.
====

==== No Empty Transitions

Connecting words must carry weight. _Furthermore_, _moreover_, _additionally_, and _also_ are warranted only when the next sentence genuinely adds something the reader couldn't infer. _Therefore_, _thus_, and _hence_ are warranted only when the next sentence follows logically from the previous and the reader benefits from the cue.

[.bad-example]
.{bad-icon} Bad example

[.good-example]

A session is created by `login`. The session is bound to one workspace and one user, both fixed for the session's lifetime. Pending changes are persisted by `save` and discarded by `rollback`.
====

=== Document Mechanics

[#section-nesting]
==== Section Nesting

A section can have any number of subsections. A subsection can nest further subsections when the material requires it. Two limits bound this nesting.

First, subsections nest at most four levels below a top-level section, so a heading is at most five levels deep counting the top-level section. In xref:#part-4[AsciiDoc], a top-level section is written `==` and each deeper level adds one `=`, making `======` the deepest permitted heading:

[source,asciidoc]
----
== 1
=== 2
==== 3
===== 4
====== 5
----

Second, a section has zero subsections, or two or more. A lone subsection means either that its content belongs in the parent section's prose or that a sibling subsection is missing.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
== Configuration
=== Connection Pool
The connection pool is configured via `pool.size`.
----

(One section, one subsection.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
== Configuration
The connection pool is configured via `pool.size`.
----

(The lone subsection's content is folded into the parent section.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
== Configuration
=== Connection Pool
The connection pool is configured via `pool.size`.
=== Cache
The cache is configured via `cache.ttl`.
----

(Two sibling subsections.)
====

[#cross-references]
==== Cross-References

A cross-reference points to a numbered section, table, figure, or defined term. References to "above" and "below" are permitted only within the same paragraph or list. Across sections, use the section number. In AsciiDoc documents, the anchor that a cross-reference targets and the markup that carries the cross-reference are governed by the sections xref:#explicit-anchors[] and xref:#internal-references-with-xref[].

[.bad-example]
.{bad-icon} Bad example
====
As discussed earlier, the resolver follows specific rules.
====

[.good-example]
.{good-icon} Good example
====
The resolver follows the rules defined in §5.3.
====

[#lists-for-enumerable-content]
==== Lists for Enumerable Content

Lists are used for content whose items are genuinely parallel and enumerable: a fixed set of states, a finite set of options, a sequence of steps. Prose is used for arguments, derivations, and connected reasoning. Don't bullet a paragraph. Don't write a list as a single run-on sentence.

[.bad-example]
.{bad-icon} Bad example
====
- The system starts.
- Then it loads the configuration.
- It also validates it and applies it in order based on priority which is a number from 1 to N.
====

[.good-example]
.{good-icon} Good example
====
The system starts in four phases. First, it reads the configuration file. Second, it validates the configuration against the schema. Third, it resolves references between configuration sections. Fourth, it applies each resolved section in ascending order of priority.
====

A bullet list is appropriate when the items are independent and parallel:

[.good-example]
.{good-icon} Good example
====
A request is in exactly one of the following states:

* RECEIVED - the request has been parsed but not yet authorized.
* AUTHORIZED - the caller has been authenticated and authorized.
* PROCESSING - the handler is executing.
* COMPLETED - the response has been sent.
* FAILED - the handler raised an unhandled error.
====

[#exhaustive-lists]
==== Exhaustive Lists

A list is either complete or explicitly marked as incomplete. When the list enumerates the members of a set, all members appear. When the list is a sample, it opens or closes with a phrase that names the omission - _among others_, _the most common are..._, _see §X for the full list_. A list that looks complete but omits members is a defect: a reader who acts on the listed items as the full set acts on a false premise.

[.bad-example]
.{bad-icon} Bad example
====
The supported authentication methods are password, OAuth, and SAML. (The system also supports certificate-based authentication and an external IdP integration, but the list reads as exhaustive. A reader who designs an integration around the three listed methods discovers the gap only during implementation.)
====

[.good-example]
.{good-icon} Good example
====
The supported authentication methods are password, OAuth, SAML, certificate-based authentication, and external IdP integration.
====

[.good-example]
.{good-icon} Good example
====
The most commonly used authentication methods are password, OAuth, and SAML. The full set, including certificate-based and external-IdP methods, is listed in §6.2.
====

==== Tables for Multidimensional Comparison

A table is used when each item has the same set of attributes and the reader will compare across attributes. A table isn't used as a layout device for text that isn't tabular. Each column has a header. Each row is independent.

[.good-example]
.{good-icon} Good example
====
[cols="^,^,^,^",options="header",]
|===
|State |Visible to caller |Counted in metrics |Eligible for retry
|RECEIVED |no |no |no
|AUTHORIZED |no |yes |no
|PROCESSING |no |yes |no
|COMPLETED |yes |yes |no
|FAILED |yes |yes |yes
|===
====

[#capitalization-of-defined-terms]
==== Capitalization of Defined Terms

A defined term is capitalized consistently with its definition. When the term is defined as _Connection Pool_, it's written _Connection Pool_ throughout, not _connection pool_ or _ConnectionPool_. When `ConnectionPool` is also a separate identifier referring to the type, both forms appear and the document distinguishes them.

[.bad-example]
.{bad-icon} Bad example
====
A _Connection Pool_ is a fixed-size set of pre-established connections. Each worker borrows a connection from the connection pool. (The term is defined as _Connection Pool_ and later written _connection pool_.)
====

[.good-example]
.{good-icon} Good example
====
A _Connection Pool_ is a fixed-size set of pre-established connections. Each worker borrows a connection from the Connection Pool.
====

[[part-2]]
== Technical Texts

This part adds rules for documents whose subject is a technical system: specifications, reference documentation, architecture documents, design documents, protocol descriptions, runbooks, and operator manuals. The rules in xref:#part-1[Part 1] still apply. xref:#part-2[Part 2] specializes them and adds rules that are specific to technical writing.

=== Show, Then Tell - Technical Form

For technical content, the illustration called for by xref:#show-then-tell[Show, Then Tell] is a concrete artifact: a code snippet, a request/response pair, a configuration file, a query, or a diagram. When the behavior has interesting boundaries - empty inputs, missing inputs, concurrent access, partial failures, error conditions - illustrate at least one boundary alongside the typical case. Bugs tend to arise at boundaries.

[.bad-example]
.{bad-icon} Bad example
====
[source,python]
----
result = foo.do_thing(bar, baz)
print(result.stuff)
----
====

[.good-example]
.{good-icon} Good example
====
[source,python]
----
order = order_service.find_by_number("ORD-2026-00471")
order_service.cancel(order, reason="customer_request")
print(order.status)  # "CANCELLED"
----
====

=== Show the Wire When Behavior is on the Wire

For network behavior, the illustration required by xref:#show-then-tell[Show, Then Tell] is the actual bytes - HTTP requests and responses, JSON payloads, SQL statements, or protocol frames - not a prose summary. The reader can then copy-paste and verify. Prose descriptions of network behavior tend to omit headers, status codes, content types, or escaping rules that turn out to matter.

[.bad-example]
.{bad-icon} Bad example
====
The client sends a POST request with the order data and receives the created order back.
====

[.good-example]
.{good-icon} Good example
====
[source,http]
----
POST /api/orders HTTP/1.1
Host: shop.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOi...

{"customer_id": "C-4711", "items": [{"sku": "SKU-001", "quantity": 2}]}
----

[source,http]
----
HTTP/1.1 201 Created
Location: /api/orders/ORD-2026-00471
Content-Type: application/json

{"order_number": "ORD-2026-00471", "status": "PENDING_PAYMENT", "total": "59.80"}
----
====

[[diagrams-and-prose]]
=== Diagrams for Structure, Prose for Behavior

Use a diagram for static structure: components, layers, topology, state transitions, and data shapes. Use prose or code for dynamic behavior: what happens, in what order, under what conditions. A diagram doesn't convey behavior that prose expresses more precisely. Prose doesn't describe a tree or graph that a diagram conveys at a glance.

[.bad-example]
.{bad-icon} Bad example
====
A six-page sequence diagram with thirty messages, each labeled with a one-line description, and no accompanying prose.
====

[.good-example]
.{good-icon} Good example
====
A component diagram showing the client, gateway, order service, and payment service as boxes with the protocols between them, accompanied by prose that describes the request flow, the failure modes, and the retry rules.
====

=== ASCII Diagrams

When the rule xref:#diagrams-and-prose[] calls for a diagram, render it as an inline ASCII diagram if the structure fits. An ASCII diagram travels with the text - it survives copy-paste, renders in plain-text contexts, and doesn't depend on an asset pipeline. Use one when the topology genuinely earns a picture: a fan-out, a layered stack, a tree, a state machine with a small number of states. Skip it when one or two sentences of prose convey the same relationship, or when the structure is too large or too irregular to render legibly in characters.

A clean ASCII diagram has aligned components, consistent connectors, and labels that name each component. Boxes are drawn with one consistent set of characters throughout the document. Each arrow points in one direction and connects two named components. A diagram that forces the reader to count spaces or trace ambiguous connections is worse than the prose it replaced.

[.bad-example]
.{bad-icon} Bad example
====
The gateway dispatches each incoming request to one of three downstream services - the search service, the catalog service, or the cart service - based on the request path. The search service queries an inverted index. The catalog service queries a relational database. The cart service queries a key-value store.

(The topology is a fan-out, and the reader has to assemble it from sequential prose. A diagram would show it at a glance.)
====

[.good-example]
.{good-icon} Good example
====
....
                    +-----------+
                    |  gateway  |
                    +-----+-----+
                          |
          +---------------+---------------+
          |               |               |
          v               v               v
    +-----------+   +-----------+   +-----------+
    |  search   |   |  catalog  |   |   cart    |
    +-----+-----+   +-----+-----+   +-----+-----+
          |               |               |
          v               v               v
    +-----------+   +-----------+   +-----------+
    |   index   |   |    db     |   |   redis   |
    +-----------+   +-----------+   +-----------+
....

Each arrow is a synchronous call. The gateway routes by request path. Downstream services don't call each other.
====

=== Self-Contained Snippets

A code snippet compiles, parses, or runs as shown. Imports, package declarations, and necessary boilerplate are either included or explicitly elided with a comment such as `// imports omitted`. Snippets that combine several concerns into a fragment that doesn't run are a hidden cost: readers spend time reconstructing the missing parts.

[.bad-example]
.{bad-icon} Bad example
====
[source,python]
----
client.send(payload)
----
====

[.good-example]
.{good-icon} Good example
====
[source,python]
----
# imports omitted
client = Client(endpoint="https://api.example.com", timeout=5.0)
payload = {"event": "user.signed_up", "user_id": "U-12345"}
response = client.send(payload)
assert response.status_code == 202
----
====

[#code-markup]
=== Code Markup

Code, identifiers, file paths, configuration keys, and protocol headers are set in a monospaced font. Inline code uses backticks: `commit`, `org.example.api`, `/etc/app.conf`, `Content-Type`. Code blocks are tagged with a language for syntax highlighting. Identifiers are written exactly as they appear in code, including case.

[.bad-example]
.{bad-icon} Bad example
====
Call commit to persist the changes.
====

[.good-example]
.{good-icon} Good example
====
Call `commit` to persist the changes.
====

[[part-3]]
== When the Author is an AI

This part adds rules that apply when an AI system is given this guideline as a style instruction. The rules in xref:#part-1[Part 1] and xref:#part-2[Part 2] still apply. xref:#part-3[Part 3] addresses failure modes specific to AI authorship: inventing facts, restating the prompt, opening with throat-clearing, narrating rule application, hedging from uncertainty, over-weighting items mentioned in the prompt, and treating the guideline's markup as content.

=== Don't Invent Facts

When you don't have a fact - an identifier, a path, a version number, a quotation, a citation, a metric, a date - state that you don't have it. Don't fabricate plausible-looking values. A reader uses the fact, and a fabricated fact becomes a defect that compounds with every downstream use.

ifdef::backend-html5[]
++++
<!-- vale Google.Acronyms = NO -->
++++
endif::[]
[.bad-example]
.{bad-icon} Bad example
====
The service runs on Postgres 16.4 with `max_connections=300` and exposes a `/health` endpoint that returns the build SHA. (None of the version, parameter, or endpoint was stated in the input.)
====
ifdef::backend-html5[]
++++
<!-- vale Google.Acronyms = YES -->
++++
endif::[]

[.good-example]
.{good-icon} Good example
====
The service runs on Postgres. The exact version and connection-pool configuration aren't stated in the input. Whether a health endpoint exists isn't documented either, and the service's route table would have to be checked to confirm.
====

[#dont-echo]
=== Don't Echo the Prompt

Begin with content. Don't open with a restatement of the request: "You asked about X. Here is information about X." The reader wrote the prompt. They don't need it read back to them.

Restating the audience and assumed knowledge to calibrate the document isn't echoing - see xref:#identify-audience[]. The prohibited pattern is summarizing the question back to the questioner before answering it.

[.bad-example]
.{bad-icon} Bad example
====
You asked about connection pooling. Here is a description of connection pooling. Connection pooling is a technique used in software systems to manage database connections.
====

[.good-example]
.{good-icon} Good example
====
A connection pool is a fixed-size set of pre-established connections that clients borrow for the duration of an operation and return when finished.
====

=== Don't Throat-Clear

ifdef::backend-html5[]
++++
<!-- vale Google.Quotes = NO -->
++++
endif::[]

This rule specializes xref:#no-throat-clearing[] for the AI failure mode. Don't open a response with service-style preambles - "Sure!", "Great question!", "I'd be happy to help.", "Of course!", "Certainly.", "Absolutely.", etc. The first sentence carries content.

++++
<!-- vale Google.Quotes = YES -->
++++
endif::[]

ifdef::backend-html5[]
++++
<!-- vale Google.Exclamation = NO -->
<!-- vale Google.FirstPerson = NO -->
++++
endif::[]
[.bad-example]
.{bad-icon} Bad example
====
Sure! I'd be happy to explain HTTP connection pooling. Let me walk you through it step by step.

HTTP connection pooling is...
====

++++
<!-- vale Google.Exclamation = YES -->
<!-- vale Google.FirstPerson = YES -->
++++
endif::[]

[.good-example]
.{good-icon} Good example
====
An HTTP connection pool keeps a fixed number of TCP connections to a remote host open across requests. The client borrows a connection for one request, returns it on completion, and the pool reuses it for the next request to the same host.
====

=== Don't Annotate Rule Application

Don't tell the reader which rule from this guideline you are applying. "Per §3.2, I'll use direct verbs here" is meta-commentary, not content. Apply the rule. Don't narrate the application.

[.bad-example]
.{bad-icon} Bad example
====
Following the guideline, I'll keep this terse and use direct verbs. Here is a description of connection pooling...
====

[.good-example]
.{good-icon} Good example
====
A connection pool is a fixed-size set of pre-established connections...
====

=== State Gaps, Don't Hedge

The xref:#no-hedging[] rule forbids softening claims with _usually_, _generally_, _somewhat_. When an AI author lacks a fact, the natural failure is to hedge: emit a softened claim that sounds plausible but conveys nothing the reader can act on. The right response is to name what's missing, not to weaken the assertion.

ifdef::backend-html5[]
++++
<!-- vale Google.Acronyms = NO -->
++++
endif::[]
[.bad-example]
.{bad-icon} Bad example
====
The cache is generally pretty fast, and the eviction policy is usually some form of LRU.
====
ifdef::backend-html5[]
++++
<!-- vale Google.Acronyms = YES -->
++++
endif::[]

[.good-example]
.{good-icon} Good example
====
The cache's latency and eviction policy aren't stated in the input. Both would have to be specified to answer the question.
====

=== Distribute by Domain, Not by Prompt

A specific element mentioned in the prompt is a minimum constraint, not a theme. The response includes that element at least once. The element's representation in the response reflects its realistic proportion in the subject domain, not its prominence in the prompt.

The response is written as if the author chose the content independently, using the prompt only to determine the topic and the minimum constraints. The distribution of content across subtopics matches the distribution a domain expert would produce when writing on that topic without a prompt.

ifdef::backend-html5[]
++++
<!-- vale Google.Acronyms = NO -->
++++
endif::[]
[.bad-example]
.{bad-icon} Bad example
====
Prompt: "List 20 Java performance optimization techniques. Include something related to garbage collection."

Response: lists 9 garbage collection techniques first (GC tuning, heap sizing, GC log analysis, G1 vs ZGC comparison...), then 11 techniques from other areas.

Garbage collection is one subcategory of JVM performance optimization. The response allocated 45% of its content to it because the prompt mentioned it explicitly, not because it represents 45% of the optimization domain.
====
ifdef::backend-html5[]
++++
<!-- vale Google.Acronyms = YES -->
++++
endif::[]
[.good-example]
.{good-icon} Good example
====
Prompt: "List 20 Java performance optimization techniques. Include something related to garbage collection."

Response: lists 20 techniques covering Just-in-Time (JIT) compilation, object pooling, connection pool sizing, lock contention reduction, I/O buffering, profiling, lazy initialization, and other areas, with 1-2 entries on garbage collection tuning placed according to natural fit.

Garbage collection appears once, satisfying the minimum constraint. Its share of the response matches its share of the optimization domain, not its share of the prompt.
====

[[part-4]]
== AsciiDoc Documents

This part adds rules for the AsciiDoc markup that carries the document. The rules in xref:#part-1[Part 1], xref:#part-2[Part 2], and xref:#part-3[Part 3] still apply. They govern the prose. This part governs the markup the prose is written in.

[#one-sentence-per-line]
=== Don't Write One Sentence Per Line

Write each paragraph as continuous prose. Don't start a new source line for every sentence. AsciiDoc joins consecutive non-blank lines into one paragraph, so a per-sentence line break produces no change in the rendered output. The source reads as a list of fragments instead of a paragraph. A sentence edit no longer aligns with the unit a reader thinks in.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
The cache stores recently used entries.
Each entry has a key and a value.
An entry is evicted when the cache is full.
----
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
The cache stores recently used entries. Each entry has a key and a value. An entry is evicted when the cache is full.
----
====

[#numbered-paragraphs]
=== Numbered Paragraphs

A _numbered paragraph_ is a paragraph written as an item of an AsciiDoc list, so that the renderer prefixes it with a number, a letter, or - at the deepest levels - a bullet. Numbering makes each paragraph an addressable unit: a review comment or a cross-reference names item 2.3 instead of "the second sentence of the fourth paragraph". A numbered paragraph is still a paragraph - full sentences developing one topic, as xref:#one-paragraph-one-topic[] requires. Numbering addresses paragraphs. It doesn't license the fragment-style bullets that xref:#lists-for-enumerable-content[] forbids.

[#number-every-paragraph]
==== Number Every Paragraph of a Multi-Paragraph Section

When the body of a section contains two or more paragraphs, every paragraph of that body is a numbered paragraph. A section whose body is a single paragraph is written as plain prose. The numbering enforces xref:#one-paragraph-one-topic[] at the markup level: each number marks the start of one topic, so a paragraph that drifts into a second topic is exposed - the second topic sits under a number that promised the first.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
== Eviction

A cache entry is evicted when its TTL expires. Eviction runs synchronously on each insertion.

The eviction order is least-recently-used. The order is tracked per shard.

Evicted entries are counted in the `cache_evictions_total` metric.
----

(Three paragraphs, none addressable. A reviewer pointing at the second one has to describe its position.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
== Eviction

. A cache entry is evicted when its TTL expires. Eviction runs synchronously on each insertion.
. The eviction order is least-recently-used. The order is tracked per shard.
. Evicted entries are counted in the `cache_evictions_total` metric.
----
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
== Eviction

A cache entry is evicted when its TTL expires. The eviction order is least-recently-used.
----

(A single-paragraph body stays plain prose. A lone number would have nothing to be ordered against.)
====

[#numbering-depth]
==== Numbering Depth

Numbered paragraphs nest at most six levels. The first four levels are ordered-list markers written with dots - `.`, `..`, `++...++`, `++....++`. The fifth and sixth levels are unordered-list markers - `++*++`, `++**++`. The renderer shows the fifth and sixth levels as bullets instead of numbers, and the other rules of this section apply to them all the same. Nothing nests below `++**++`:

[source,asciidoc]
----
. 1
.. 2
... 3
.... 4
* 5
** 6
----

Material that seems to need a seventh level doesn't get one. The subtree is restructured instead: the top of the subtree is promoted to a subsection heading per xref:#section-nesting[], or intermediate levels that don't carry real hierarchy are flattened.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
.... The client retries a failed request.
..... The retry delay doubles on each attempt.
*** The delay is capped at 30 seconds.
----

(Five dots exceed the four-dot ladder, and `++***++` exceeds the two-asterisk ladder. After `++....++` come `++*++` and `++**++`, and nothing comes after `++**++`.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
.... The client retries a failed request.
* The retry delay doubles on each attempt.
* The delay is capped at 30 seconds.
----
====

[#no-lone-numbered-paragraph]
==== No Lone Numbered Paragraph

At every nesting level, a numbered paragraph has at least one sibling at the same level under the same parent. A level holds zero items or two or more. A lone item means the numbering does no work: there is nothing at that level to order the item against, so its content belongs in the parent paragraph's own text or in a continuation per xref:#continuation-content[] - or a sibling is missing. The rule mirrors xref:#section-nesting[], which forbids a lone subsection for the same reason.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
. The pool validates a connection before lending it.
.. The validation query is `SELECT 1`.
. The pool replaces a connection that fails validation.
----

(The `..` item has no sibling. It elaborates its parent rather than enumerating anything.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
. The pool validates a connection before lending it. The validation query is `SELECT 1`.
. The pool replaces a connection that fails validation.
----

(The lone item is folded into its parent.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
. The pool validates a connection before lending it.
.. The validation query is `SELECT 1`.
.. A validation that exceeds 250 ms counts as a failure.
. The pool replaces a connection that fails validation.
----

(The missing sibling is supplied, so the nested level now enumerates two facts about validation.)
====

[#paragraph-headers]
==== Paragraph Headers

A numbered paragraph may open with a _paragraph header_ - a bold phrase that names the paragraph's topic. The header stands alone on the item's line. The paragraph's text follows as a continuation per xref:#continuation-content[] or as nested numbered paragraphs. A header is due when it helps the reader scan: the paragraph is long and holds several sentences, or the item heads a group of nested items whose shared topic needs a name. A header is omitted when it adds nothing: a paragraph of one or two sentences whose topic is already stated by the section heading doesn't restate that topic in bold. A paragraph header is the one place where bold appears in body text - see xref:#inline-formatting-semantics[].

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
== Dispatcher

. The dispatcher is an Apache HTTP module placed in front of the rendering instances.
. *Security*
+
The dispatcher filters incoming requests and forwards only those that match the configured allowlist of paths and methods. A request outside the allowlist is rejected with status `404`. The allowlist is defined in `dispatcher/filters.any` and is deployed with the application code.
. *Caching*
+
The dispatcher stores rendered pages on disk. On a cache miss, the request is forwarded to a rendering instance and the response is stored before being returned. A publish event invalidates the affected paths.
----

(The first paragraph is short and needs no header. The other two are long, and the headers let the reader locate the security and caching material without reading both paragraphs.)
====

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
== Defaults

. *Default pool size* The default pool size is 10.
. *Default timeout* The default timeout is 30 seconds.
----

(Each header restates the one sentence it precedes, under a heading that already names the topic. The headers add bold without adding information.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
== Defaults

. The default pool size is 10.
. The default timeout is 30 seconds.
----
====

[#continuation-content]
==== Continuation Content

Content whose meaning is strongly bound to the preceding numbered paragraph is attached to that paragraph with a list continuation - a line holding a single `+` - instead of being numbered. The attached block can be a paragraph that elaborates the item, a code listing, an image, a table, or an admonition. The continuation signals that the item's topic continues. Content that starts a new topic isn't attached - it takes its own number.

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
------
. The pool is configured via `pool.size`:
+
[source,yaml]
----
pool:
  size: 25
----
+
The value is read once at startup. A change requires a restart.
. A connection that fails validation is evicted and replaced.
------

(The listing and the follow-up paragraph elaborate the configuration item, so both attach to it with `+`. The eviction fact starts a new topic and takes its own number.)
====

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
. The pool is configured via `pool.size`.
+
The cache is configured via `cache.ttl`.
----

(The attached paragraph starts a new topic. The cache isn't an elaboration of the pool's configuration, so the paragraph takes its own number.)
====

A continuation paragraph followed by deeper-level numbered paragraphs needs one more `{plus}` line between the paragraph and the first deeper item. That `{plus}` ends the continuation paragraph and attaches the deeper items to the same parent. Without it, the renderer doesn't recognize the deeper item's marker: it joins the marker and the item's text into the continuation paragraph as plain text and renders the item's own `{plus}` lines literally. A following item at the same level needs no such `{plus}` - its marker ends the continuation paragraph on its own.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
. The pool validates a connection before lending it:
+
Validation runs on every lease, not only on creation.
.. The validation query is `SELECT 1`.
.. A validation that exceeds 250 ms counts as a failure.
----

(The `..` markers directly follow the continuation paragraph. The renderer outputs one paragraph reading "Validation runs on every lease, not only on creation. .. The validation query is `SELECT 1`. .. A validation that exceeds 250 ms counts as a failure.")
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
. The pool validates a connection before lending it:
+
Validation runs on every lease, not only on creation.
+
.. The validation query is `SELECT 1`.
.. A validation that exceeds 250 ms counts as a failure.
----
====

[#explicit-anchors]
=== Explicit Anchors

A part of the text that a cross-reference targets carries an explicit anchor: a block anchor `+[[kebab-case-id]]+` on the line directly above a heading, or an inline anchor of the same form placed directly on text inside a block. The anchor sits on the most specific part of the text that defines the referenced concept - the section when the section as a whole defines it, the paragraph or list item when that alone defines it, or the single defining word. An anchor ID is a lowercase kebab-case name in the document's language, and it names the content, not the content's position. An explicit ID survives retitling. An auto-generated ID is derived from the heading text, so renaming the heading silently breaks every reference to it.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
== Configuration

The defaults are listed in <<_connection_pool>>.

=== Connection Pool
----

(The reference targets the auto-generated ID. Renaming the heading to "Pooling" breaks the reference.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
== Configuration

The defaults are listed in the section xref:#connection-pool[].

[[connection-pool]]
=== Connection Pool

A [[lease]]_lease_ is the exclusive right to use one pooled connection for the duration of an operation. [[pool-size-default]]The default pool size is 10.
----

(The section keeps its ID under any title. The anchors sit on the most specific defining parts: the section for the pool, the word for the lease, the statement for the default size.)
====

[#internal-references-with-xref]
=== Internal References with `xref`

A part of the text that refers to a concept defined in another part of the text links to the definition. The link is written with the `xref` macro against the definition's explicit anchor. The default form is `+xref:#connection-pool[]+` with empty brackets, which renders the target's title and stays correct when the title changes. Custom link text is supplied only when the title doesn't fit the grammar of the sentence. When a reference points to a section and the rendered title alone would leave the grammar or the meaning of the sentence unclear, the reference is preceded by a qualifier such as _the section_ or _the chapter_. xref:#cross-references[] governs the wording around the reference. This rule governs when a reference is due and the markup that carries it.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
Each handler holds a lease on a pooled connection. Pool sizing is described in the section "Connection Pool" above. Defaults are listed in xref:#connection-pool[].
----

(The term _lease_ is defined elsewhere in the document but isn't linked to its definition. The first section reference is plain text: it doesn't link, and it breaks when the section moves or is retitled. The last reference links, but it renders as "listed in Connection Pool" - without a qualifier the meaning of the sentence is unclear.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
Each handler holds a xref:#lease[lease] on a pooled connection. Pool sizing is described in the section xref:#connection-pool[]. Size the pool according to the xref:#connection-pool[sizing rules].
----

(The defined term links to its definition. The empty-bracket form inherits the target's title, and the qualifier _the section_ keeps the meaning of the sentence clear. The last reference supplies custom text because the sentence needs a different inflection.)
====

The empty-bracket form takes its text from the target's reference text. A section supplies its title. Any other anchor - on a paragraph, a list item, or a word - supplies a reference text only when the anchor declares one, in the form `+[[anchor-id,reference text]]+`. An empty-bracket reference against an anchor without a reference text renders the raw anchor ID, which is a defect. When the target isn't a section, either declare the reference text on the anchor or supply custom link text in the reference.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
[[retention-schedule]]The retention schedule keeps records for 90 days.

The purge job applies xref:#retention-schedule[].
----

(The anchor sits on a paragraph and declares no reference text, so the reference renders as the raw ID: "The purge job applies [retention-schedule].")
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
[[retention-schedule,the retention schedule]]The retention schedule keeps records for 90 days.

The purge job applies xref:#retention-schedule[].
----

(The anchor declares a reference text, so the reference renders as "The purge job applies the retention schedule.")
====

[#link-text-carries-the-claim]
=== Link Text Carries the Claim

An external link wraps the phrase that the linked source substantiates, so the sentence keeps its meaning when the markup is stripped. Link text such as _here_ or _this page_, and a bare URL pasted mid-sentence, carries no information about the target. A link that leads the reader away from the document ends with `^`, which opens the target in a new tab.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
The upload limit is documented here: https://docs.example.com/assets/limits.
----
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
The service https://docs.example.com/assets/limits[accepts uploads of up to 5 GB per file^].
----

(The linked phrase is the claim the source substantiates. The sentence keeps its meaning without the markup.)
====

[#inline-formatting-semantics]
=== Inline Formatting Semantics

Each inline mark has one role. A monospaced span marks technical literals - paths, domains, IP addresses, file extensions, configuration keys, HTTP headers, identifiers - as xref:#code-markup[] prescribes. Italics mark a term of art, a foreign-language term, the proper name of a product or interface element, a word mentioned as a word, and a word that carries emphasis. Bold appears in headings, where the renderer applies it, and in the paragraph headers prescribed by xref:#paragraph-headers[]. Elsewhere, bold isn't written in body text - a word that needs emphasis in a paragraph is italicized.

[.bad-example]
.{bad-icon} Bad example
====
[source,asciidoc]
----
The *connection string* is stored in _config.yaml_ under the key *db.url*.
----

(Bold in body text - twice - and italics for a path. Emphasis belongs to italics, and paths belong to monospaced spans.)
====

[.good-example]
.{good-icon} Good example
====
[source,asciidoc]
----
A _connection string_ is a URL-shaped value that names the database host, port, and schema. The connection string is stored in `config.yaml` under the key `db.url`. The value follows the _JDBC_ URL convention.
----
